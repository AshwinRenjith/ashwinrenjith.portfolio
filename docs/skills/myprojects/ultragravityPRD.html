01 Vision & Mission

Build the operating system
for the agentic era.

OpenClaw proved the market. It showed the world that a personal AI agent, running locally, accessible through any messaging app, is something millions of people want. UltraGravity is what OpenClaw would have been if it were designed by a team who obsessed over performance, security, and developer ergonomics from day one.

UltraGravity is not a "chatbot framework." It is a high-performance agentic runtime — a single binary (or Docker image) that manages sessions, memory, tool execution, channel normalization, multi-agent orchestration, and security policy enforcement, all in a way that is orders of magnitude faster and safer than any existing solution.

Core Thesis

The bottleneck in personal AI agents is not the LLM. It is the infrastructure around the LLM — the queue management, the context assembly, the tool sandboxing, the memory retrieval, and the security enforcement. UltraGravity solves all of these at the systems level, in Rust, so the developer never has to think about them.

Design Principles

⚡

Speed First

Every architectural decision is evaluated against its latency impact. The gateway core is written in Rust. The hot path from message receipt to LLM invocation must be under 40ms, excluding network round-trips.

🔒

Secure by Default

Security is not a layer on top — it is the foundation. Every tool execution runs in an isolated process with seccomp/Landlock. Prompt injection is structurally impossible at the message routing layer.

🧩

Zero-Config Extensibility

New channels, tools, memory backends, and LLM providers drop in as self-contained packages with no modifications to core code. Discovery is automatic. Skills hot-reload in under 200ms.

🌐

Multi-Tenant Ready

Unlike OpenClaw's personal-first design, UltraGravity is built from the ground up for team and enterprise deployment with per-workspace quotas, audit logs, and role-based access control.

📡

LLM Agnostic

First-class support for Claude, GPT-4o, Gemini, Mistral, Llama, and any OpenAI-compatible endpoint. Automatic fallback routing and cost-aware model selection built in.

🧠

Intelligent Memory

Hybrid memory architecture: vector similarity search, BM25 keyword search, and a novel temporal decay model that surfaces the most contextually relevant memories — not just the most recent.

LLM Provider Router

Unlike OpenClaw which defaults to Claude and has manual provider switching, GravityCore's LLM Router implements automatic intelligent routing based on multiple signals:

  • Task type classification — code tasks route to models with strong coding benchmarks; reasoning tasks to models with strong chain-of-thought.
  • Cost optimization — user-configurable budget thresholds automatically fall back to cheaper models for low-complexity turns.
  • Latency optimization — if a provider's P50 response time has spiked in the last 60 seconds, the router shifts traffic to the backup provider.
  • Failover — automatic retry with exponential backoff, then provider failover on 5xx errors.
GravityCore — LLM Router (Conceptual)
Rust
// LLM Router — selects optimal provider per turn
pub struct LLMRouter {
    providers: Vec<Box<dyn LLMProvider>>,
    health: Arc<RwLock<HealthMap>>,
    policy: RouterPolicy,
}

impl LLMRouter {
    pub async fn route(&self, ctx: &TurnContext) -> Result<LLMStream> {
        let candidate = self.policy.select(
            &self.providers,
            &self.health.read().await,
            &ctx.task_class,   // code | reasoning | creative | simple
            ctx.budget_remaining,
            ctx.latency_target,
        );

        match candidate.invoke_stream(ctx).await {
            Ok(stream) => Ok(stream),
            Err(e) if e.is_retriable() => {
                // Mark provider unhealthy, retry with fallback
                self.health.write().await.mark_degraded(candidate.id);
                self.route_fallback(ctx).await
            }
            Err(e) => Err(e),
        }
    }
}

Context Assembly — Parallel by Design

OpenClaw assembles context sequentially: load history, then search memory, then select skills. Each step blocks on the previous. UltraGravity fires all three in parallel using Tokio's join! macro and fuses the results:

Parallel Context Assembly
Rust
pub async fn assemble_context(
    session: &Session,
    query: &str,
    mem: &GravityMem,
    skills: &SkillRegistry,
) -> Context {

    // All three fire simultaneously — no sequential blocking
    let (history, memories, selected_skills) = tokio::join!(
        session.recent_history(32),
        mem.hybrid_recall(query, 8),    // vector + BM25 fused
        skills.pre_select(query),         // semantic skill match
    );

    Context::build()
        .system(session.workspace.system_prompt())
        .memories(memories)
        .skills(selected_skills)
        .history(history)
        .user(query)
        .seal()  // immutable after build — security property
}
06 Concurrency Engine

WarpQueue — Beyond the Lane Queue

OpenClaw's Lane Queue is clever: serial execution per session eliminates race conditions. But it's built on JavaScript's single-threaded event loop, which means a CPU-intensive tool execution can still stall other sessions. WarpQueue is the Rust-native evolution of this idea.

How WarpQueue Improves on the Lane Queue

🏎️

Tokio Async Core

Built on Tokio, the production-grade Rust async runtime. A blocked tool task doesn't block the event loop — it yields. Thousands of sessions can be in flight simultaneously on a single machine.

📊

Priority Levels

Each message is assigned a priority: URGENT (interrupts), NORMAL (conversational), BACKGROUND (cron, memory extraction). URGENT messages can preempt BACKGROUND tasks cooperatively.

⚡

Subtask Lanes

Within a single turn, parallel subtasks (e.g., a web search + a file read happening simultaneously) execute in dedicated lanes without blocking the main conversation queue.

🚫

Cooperative Cancellation

Every task receives a CancellationToken. When the user sends "stop" or a timeout fires, the cancellation signal propagates atomically through all in-flight tool calls and sub-agents.

Queue Architecture

WarpQueue — Core Data Structure
Rust
pub struct WarpQueue {
    // Per-session queue — keyed by workspace:channel:user
    sessions: DashMap<SessionKey, SessionQueue>,
    semaphore: Arc<Semaphore>,  // global concurrency cap
}

pub struct SessionQueue {
    primary: BinaryHeap<PriorityTask>,   // priority-ordered turns
    background: VecDeque<Task>,          // cron, webhooks, memory ops
    active: Option<TaskHandle>,          // currently running turn
    cancel: CancellationToken,
}

impl WarpQueue {
    pub async fn enqueue(
        &self,
        key: SessionKey,
        task: Task,
        priority: Priority,
    ) {
        let mut q = self.sessions.entry(key).or_default();
        if priority == Priority::Urgent {
            // Cooperatively cancel current background task if running
            if let Some(active) = &q.active {
                if active.priority < Priority::Normal {
                    q.cancel.cancel();
                }
            }
        }
        q.primary.push(PriorityTask { task, priority });
        self.try_run_next(&mut q).await;
    }
}
Why This Matters

In OpenClaw, if a bash tool call takes 5 seconds, that session is blocked for 5 seconds. In UltraGravity, the async runtime yields control back immediately — other sessions, background memory operations, and health checks all continue uninterrupted. At 10,000 concurrent sessions, this difference is the line between a product that works and one that collapses under load.

08 Intelligence Layer

The Intelligence Layer

The intelligence layer is where UltraGravity earns its name. It combines a native agent loop, a semantic skill engine, a multi-model router, and proactive heartbeat capabilities — all without delegating to external frameworks.

System Prompt Architecture

The system prompt is never a single string. It is a layered, typed data structure assembled by GravityCore at each turn, with strict content-source labeling. Each layer has a defined trust class, and the LLM is instructed (at the structural level) about which layers are user-influenced vs. system-authoritative.

SYSTEM LAYER 0: GravityCore Rules IMMUTABLE
SYSTEM LAYER 1: Workspace Config (AGENT.md / SOUL.md) ADMIN-CONTROLLED
CONTEXT LAYER 2: Active Skills (injected, labelled) SKILL-SYSTEM
CONTEXT LAYER 3: Retrieved Memories (labelled as memories) MEMORY-SYSTEM
HISTORY LAYER 4: Conversation history (turn-by-turn) HISTORY
USER LAYER 5: Current user message [UNTRUSTED-USER-INPUT] USER-INPUT
Structural Prompt Injection Defense

The [UNTRUSTED-USER-INPUT] label is prepended to every user message and tool result at the structural level before model invocation. The GravityCore rules (Layer 0) explicitly instruct the model never to treat UNTRUSTED-USER-INPUT as instructions to change behavior. This is not a heuristic — it is an architectural constraint on context assembly that applies to every single turn.

Heartbeat Daemon

UltraGravity's heartbeat daemon is a proactive agent scheduler that runs independently of user messages. It wakes on configurable schedules and evaluates whether to take autonomous actions:

  • Digest generation — Morning summaries of pending tasks, calendar events, unread items.
  • Memory consolidation — Periodically compresses conversation history into durable memory entries.
  • Proactive alerts — If a monitored metric (price, news keyword, calendar event) crosses a threshold, the agent messages the user autonomously.
  • Cron-based skill execution — Skills can declare schedule: "0 9 * * *" in their manifest to trigger automatically.

GravityGate — The Network Layer

GravityGate is the single network entry point for the entire runtime. It is built on axum (Rust's fastest HTTP framework, built on Tokio/Hyper) and handles all inbound connections regardless of protocol:

  • HTTP/2 REST API — OpenAI-compatible /v1/chat/completions endpoint for third-party tool integration.
  • WebSocket control channel — The Web UI and CLI maintain a persistent WS connection for real-time session streaming and control commands.
  • Webhook receiver — /webhooks/:channel receives events from Telegram, Slack, Discord, Twilio, etc. HMAC signature verification happens before any payload is parsed.
  • Server-Sent Events — Token streaming to web clients uses SSE rather than WebSockets for simpler client-side handling and proxy compatibility.

10 Memory Architecture

GravityMem — Hybrid Memory Engine

Memory is what makes an AI agent feel genuinely intelligent over time. GravityMem is a purpose-built memory system that goes far beyond what OpenClaw offers, combining four retrieval strategies into a single fused result.

Four Retrieval Strategies

🔢

ANN Vector Search

HNSW (Hierarchical Navigable Small World) index built on top of SQLite using a native Rust implementation. Sub-millisecond approximate nearest neighbor search across hundreds of thousands of memory entries.

🔤

BM25 Full-Text Search

SQLite FTS5 with BM25 ranking. Catches exact keyword matches that vector search misses, especially for proper nouns, code identifiers, and specific dates.

⏰

Temporal Decay Model

Every memory entry has an "importance score" that decays exponentially over time unless accessed. Recently accessed or reinforced memories surface higher. Configurable half-life (default: 30 days).

🕸️

Graph Memory (Entity Links)

New vs. OpenClaw. Entity extraction builds a lightweight knowledge graph (person → organization, project → deadline, etc.). Queries can traverse graph edges to pull in related context automatically.

Memory Entry Schema

GravityMem — Memory Entry
Rust
pub struct MemoryEntry {
    pub id: Uuid,
    pub workspace: WorkspaceId,
    pub session_key: Option<SessionKey>,   // None = workspace-global
    pub content: String,                   // Human-readable text
    pub embedding: Vec<f32>,               // Dimension: 1024 (Mistral) or 384 (local)
    pub entities: Vec<Entity>,             // Extracted named entities
    pub importance: f32,                    // 0.0–1.0, decays over time
    pub access_count: u32,                 // Reinforcement tracking
    pub created_at: DateTime<Utc>,
    pub last_accessed: DateTime<Utc>,
    pub source: MemorySource,              // Conversation | Skill | External | Manual
    pub tags: Vec<String>,
}

// Temporal decay formula (applied at read time, not write time)
fn effective_score(entry: &MemoryEntry, now: DateTime<Utc>) -> f32 {
    let age_days = (now - entry.last_accessed).num_days() as f32;
    let decay = (-age_days / HALF_LIFE_DAYS).exp2();
    entry.importance * decay * (1.0 + entry.access_count as f32 * 0.1)
}

Hybrid Recall — Score Fusion

The four retrieval strategies each return a ranked list. GravityMem fuses these using Reciprocal Rank Fusion (RRF), a well-studied algorithm that combines multiple ranked lists without requiring score normalization:

Reciprocal Rank Fusion — Score Fusion
Rust
pub fn rrf_fuse(
    vector_ranks: &[Ranked<Uuid>],
    bm25_ranks: &[Ranked<Uuid>],
    temporal_ranks: &[Ranked<Uuid>],
    k: f32,  // typically 60.0
) -> Vec<Uuid> {
    let mut scores: HashMap<Uuid, f32> = HashMap::new();

    for list in [vector_ranks, bm25_ranks, temporal_ranks] {
        for (rank, item) in list.iter().enumerate() {
            *scores.entry(item.id).or_default() += 1.0 / (k + rank as f32 + 1.0);
        }
    }

    let mut sorted: Vec<_> = scores.into_iter().collect();
    sorted.sort_by(|a, b| b.1.partial_cmp(&a.1).unwrap());
    sorted.into_iter().map(|(id, _)| id).collect()
}

Memory Files — Human-Readable First

Following OpenClaw's excellent practice, all user-facing memory is additionally exported as Markdown files in ~/.gravity/workspace/memories/. You can open them in any text editor, edit them, and the changes sync back into the vector index on next startup. Memory is never a black box.

# GitHub PR Reviewer

When the user asks to review a pull request, follow this workflow:

1. Extract the PR URL or number from the message
2. Use `gh pr view {number} --json files,title,body,additions,deletions` to fetch PR data
3. For each changed file, analyze: purpose of change, potential bugs, style issues
4. Summarize in the format: **Summary** → **Concerns** → **Suggestions**

## Output Format
Always use structured markdown with severity labels: 🔴 Critical, 🟡 Warning, 🟢 Suggestion

The Skill Graph

At startup and on hot-reload, GravitySkills compiles all SKILL.md files into a Skill Graph — a vector-indexed, in-memory data structure that enables sub-5ms semantic skill pre-selection. Only skill names, descriptions, and trigger phrases are indexed at startup (~97 bytes per skill). Full content is loaded only when a skill is selected for a turn.

Self-Authoring Skills

One of UltraGravity's most powerful features: the agent can write and deploy new skills within a conversation. The user says "create a skill for tracking my wine collection" and the agent:

  1. Generates a complete, valid SKILL.md file with proper frontmatter
  2. Writes it to the skills directory via the File tool (sandboxed to skills path only)
  3. The GravitySkills watcher detects the file change via inotify/kqueue
  4. Recompiles the skill graph — within 200ms the skill is live
  5. Confirms to the user with the skill's semantic triggers
12 Multi-Agent System

GravityMesh — Native Multi-Agent

OpenClaw has basic multi-agent session tools. UltraGravity ships a full agent mesh with capability negotiation, parallel execution, and streamed result aggregation — all within the same runtime.

Agent Roles

🎯

Orchestrator Agent

The user-facing agent. Receives the user's message, breaks it into subtasks, dispatches to specialist agents, aggregates results, and delivers the unified response. Never executes tools directly.

⚙️

Specialist Agents

Spawned on-demand per subtask. Have a constrained tool set (e.g., a "WebSearch Agent" only has browser access). Operate in parallel. Results streamed back to the orchestrator as they complete.

👀

Observer Agents

Long-running background agents that monitor external sources (RSS feeds, APIs, price trackers) and push events into the orchestrator's queue when thresholds are crossed. Persistent heartbeat mode.

Execution Model

Multi-Agent Parallel Dispatch
Rust
// Orchestrator decomposes task and fires specialists in parallel
pub async fn orchestrate(
    task: &str,
    mesh: &GravityMesh,
) -> OrchestratorResult {

    // LLM decomposes into subtasks with capability requirements
    let subtasks = decompose(task).await;

    // Spawn specialists in parallel — results stream as they complete
    let mut handles = JoinSet::new();
    for subtask in subtasks {
        let agent = mesh.spawn_specialist(&subtask.required_caps).await;
        handles.spawn(async move {
            agent.execute(subtask).await
        });
    }

    // Aggregate results as they stream in
    let mut results = Vec::new();
    while let Some(r) = handles.join_next().await {
        results.push(r?);
    }

    synthesize(results).await  // LLM synthesizes final response
}
13 End-to-End Workflow

Complete Message Lifecycle

Tracing a single WhatsApp message from receipt to response delivery, with timing targets at each stage.

T+0ms — Ingestion
WhatsApp WebSocket Event Received
The WhatsApp channel adapter (L1) receives a messages.upsert event from Baileys. Raw message decoded, sender identified, media downloaded if present. Message normalized to InboundMessage with TrustLevel::Paired.
T+1ms — Auth & Access Control
GravityGate Authorization Check
Sender's phone number checked against workspace allowlist. DM policy evaluated. If pairing required and not yet paired, pairing flow initiated and message rejected. All checks O(1) via pre-loaded hash map.
T+2ms — Queue Enqueue
WarpQueue — Session Key Resolution + Enqueue
Session key workspace:whatsapp:+1234567890 computed. Task enqueued with Priority::Normal. If session is idle, task starts immediately. If session is GENERATING, task queues behind current turn. No blocking.
T+3ms — Parallel Retrieval (fired simultaneously)
GravityMem Hybrid Recall + History Load + Skill Select
Three async operations fire in parallel via tokio::join!: (1) ANN vector search + BM25 + temporal re-rank in GravityMem, (2) recent session history loaded from SQLite WAL, (3) semantic skill pre-selection against compiled skill graph. All three complete within ~15ms total.
T+18ms — Context Sealed
GravityCore Context Assembly Complete
Parallel retrieval results fused. Context object assembled with all 6 layers (GravityCore rules → workspace config → skills → memories → history → user message). Context sealed and immutable. GravityShield injects [UNTRUSTED-USER-INPUT] label on user content.
T+20ms — LLM Invoked
LLM Router Selects Provider — Streaming Begins
LLM Router evaluates provider health and cost budget. Selects Claude Sonnet 4 (default). Streaming POST to Anthropic API. First token typically arrives in 300–800ms. Streamed tokens forwarded to WhatsApp adapter in near-real time (typing indicator active).
T+20ms+tool_time — Tool Execution (if needed)
GravityShield Sandbox — Tool Dispatch
If the model emits tool calls, GravityShield validates each tool call against the session's capability bitmask. Approved calls dispatched to sandboxed child processes (Landlock + seccomp). Results returned as tool result blocks, appended to context, model continues streaming.
T+stream_end — Response Delivery
WhatsApp Adapter — Format + Send
Complete response formatted for WhatsApp: markdown normalized to WA-compatible format, long messages chunked at paragraph boundaries, media files uploaded separately. Sent via Baileys sendMessage.
T+stream_end+async — Persistence (Non-blocking)
GravityMem Background Write
Turn persisted to SQLite WAL. Memory extraction runs asynchronously: entity extraction, importance scoring, embedding generation, HNSW index insertion. User response is never delayed waiting for this. Audit log entry written. OpenTelemetry span closed.
14 Security Architecture

GravityShield — Defense in Depth

OpenClaw's security is "defensible but shallow." GravityShield implements six independent security layers, each of which stops a distinct class of attacks. Bypassing any single layer requires defeating all inner layers too.

L1
Network Perimeter
Loopback-only bind by default. TLS enforced for all non-loopback. HMAC webhook signature verification before payload parse. No publicly accessible ports unless explicitly configured with token auth.
L2
Authentication & Identity
Per-channel allowlists. Device pairing protocol with QR + TOTP for CLI access. JWT tokens with 1-hour expiry for API access. No anonymous access to any endpoint by default. Rate limiting per identity, not per IP.
L3
Trust Level System
Four trust levels: Owner, Trusted, Paired, Guest. Trust level assigned by the channel adapter and immutable at the message struct level. Trust level governs which tools, which skills, and which capabilities are available. Never derived from message content.
L4
Structural Prompt Injection Firewall
All user-sourced content (messages, tool results, web scrapes) is labeled [UNTRUSTED-USER-INPUT] at the context assembly level. GravityCore rules (injected as Layer 0, never visible to user) explicitly instruct the model to treat this label as a hard trust boundary. Architecturally prevents role-escalation injection.
L5
Tool Execution Sandbox
Every tool execution spawns an isolated child process. Linux: Landlock filesystem restrictions + seccomp-bpf syscall filter. macOS: App Sandbox + file quarantine. Capability bitmask (read_fs, write_fs, network, exec) granted per-session based on trust level. No tool can exceed its declared capability surface.
L6
Audit Log & Content Policy
Every tool call, model invocation, skill activation, and trust decision is written to a tamper-evident append-only audit log (JSONL with chained hashes). Content policy engine evaluates LLM outputs before delivery. Configurable output filters for PII, sensitive credentials, and custom regexes.

16 Tool Security

Tool Sandboxing — Landlock + seccomp

UltraGravity's tool execution is hardened at the operating system level. This is not something OpenClaw does at all — it runs tool processes with the full permissions of the parent Node.js process.

Capability Bitmask System

Tool Capability Declaration
Rust
bitflags! {
    pub struct ToolCapabilities: u32 {
        const READ_FS_WORKSPACE  = 0b00000001;   // read workspace dir only
        const WRITE_FS_WORKSPACE = 0b00000010;   // write workspace dir only
        const READ_FS_HOME       = 0b00000100;   // read user home dir
        const WRITE_FS_HOME      = 0b00001000;   // write user home dir
        const EXEC_SUBPROCESS    = 0b00010000;   // spawn child processes
        const NETWORK_OUTBOUND   = 0b00100000;   // outbound TCP/UDP
        const NETWORK_LISTEN     = 0b01000000;   // bind to local port
        const READ_CREDENTIALS   = 0b10000000;   // access credentials store
    }
}

// Trust level → capability grant mapping
impl TrustLevel {
    pub fn default_caps(&self) -> ToolCapabilities {
        match self {
            Owner   => ToolCapabilities::all(),
            Trusted => READ_FS_WORKSPACE | WRITE_FS_WORKSPACE | NETWORK_OUTBOUND,
            Paired  => READ_FS_WORKSPACE | NETWORK_OUTBOUND,
            Guest   => ToolCapabilities::empty(),  // no tools at all
        }
    }
}
Linux Landlock Implementation

On Linux, each tool subprocess is spawned with a Landlock ruleset applied before the subprocess's code runs. The ruleset allows only the directories corresponding to the granted capability flags. A READ_FS_WORKSPACE grant allows O_RDONLY on ~/.gravity/workspace/ and nothing else. seccomp-bpf then further filters the allowed syscalls to a minimal set (read, write, open, stat, close, mmap, munmap, exit). Even if a tool's code is compromised, it cannot exfiltrate data, spawn network connections, or write outside its permitted paths.
